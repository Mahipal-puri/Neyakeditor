# Feature: AI styles card — Ghibli, Anime, Cartoon, Sketch

**Date:** 2026-05-28
**Kind:** Feature
**Area:** Editor right panel + dev-server proxy

## Summary

New "AI styles" card in the editor's right panel with four one-click presets — 🍃 Ghibli, ✨ Anime, 🎨 Cartoon, ✏️ Sketch — that produce a styled merge-layer overlay on the current base image. Each preset has both an AI prompt (sent to Replicate when configured) and a tuned CSS-filter approximation (used as fallback when Replicate isn't configured), so the buttons always do something visible.

Behaviour adapts to env:

- **`REPLICATE_STYLE_MODEL` set** → editor POSTs the baked-with-current-edits PNG plus the preset prompt to `/api/stylize`; the model's output becomes an opaque merge layer (`opacity: 1`, `blend: normal`).
- **Unset (default)** → editor renders the base image through the preset's CSS-filter combo into a canvas, sets the result as the merge layer with the preset's tuned `blend` + `opacity` (Ghibli=soft-light/0.75, Anime=overlay/0.6, Cartoon=screen/0.55, Sketch=multiply/0.85). Instant, free, "filter approximation" look.

The card's subtitle flips between "Replicate configured · 10–30 s per style" and "Filter approximation · set REPLICATE_STYLE_MODEL …" based on a one-shot GET to `/api/stylize` at mount.

## Files changed

- [client/src/pages/Editor.jsx](../client/src/pages/Editor.jsx) — new `STYLE_PRESETS` constant; new `stylizing` and `stylizeConfigured` state; mount-time GET to `/api/stylize`; new `applyStyle(preset)` async function; new AI Styles card UI rendered between Presets and Merge layer.
- [client/vite.config.js](../client/vite.config.js) — new `replicateStylize(env)` plugin handling `GET /api/stylize` (probe) and `POST /api/stylize` (image + prompt proxy to Replicate); registered in the `plugins` array.
- [client/.env.example](../client/.env.example) — new `REPLICATE_STYLE_MODEL=` block documenting how to enable the AI path and listing example img2img model names.

## Behavior

- Card disabled until a base image is loaded.
- Each button shows a spinner + "Applying…" while in flight; only one preset can run at a time.
- Result always lands on the merge layer — never overwrites the base image — so the user can adjust opacity / blend / clear via the existing Merge layer card.
- Errors surface as warning toasts (missing token, missing model, Replicate API failures).

## How to enable the Replicate path

1. Set `REPLICATE_API_TOKEN` and `REPLICATE_STYLE_MODEL` in `client/.env`. Example model names live in [client/.env.example](../client/.env.example).
2. Save — Vite auto-restarts on config change (look for "server restarted" in the dev terminal).
3. The AI styles card subtitle flips to "Replicate configured · 10–30 s per style" on next page load.

## Verification

1. Without `REPLICATE_STYLE_MODEL`: open `/editor`, upload a base image, click any of the four style buttons. A merge-layer overlay appears immediately with the preset's filter + blend; the existing Merge layer Opacity slider and Blend dropdown adjust it.
2. With `REPLICATE_STYLE_MODEL` and `REPLICATE_API_TOKEN` set: click any button — toast confirms "Applying X via Replicate — usually 10–30 seconds"; on success the AI-generated styled image becomes the merge layer (opaque, normal blend).
3. Click Remove on the Merge layer card — styled overlay clears, base image stays.

## Amendment — 2026-05-28: k-means adaptive palettes + inception neural model + neural post-processing

Three quality bumps stacked in one round, after user feedback that the existing local + neural results still didn't pop hard enough.

### K-means adaptive color quantization

Replaced the hand-curated fixed palettes (Ghibli warm pastels, Anime vivid cels, Cartoon comic primaries) with **per-image k-means clustering**. New helpers in [stylize.js](../client/src/lib/stylize.js):
- `kmeansPalette(imgData, k=8, maxIter=12)` — extracts an adaptive RGB palette by clustering ~10k uniformly-sampled pixels. Subsampling keeps it fast (~60–120 ms on 1 Mpx). Init uses luminance-sorted even spacing so the centroids reliably cover the dynamic range instead of converging to similar colors.
- `shiftPalette(palette, { hueShift, satMul, lightMul })` — operates in HSL space so we can take an image-derived palette and *nudge* it toward a style feel (warm-shift + slight desat for Ghibli pastel; saturation boost for Anime cels; mild boost for Cartoon).
- `rgbToHsl` / `hslToRgb` helpers added for the above.

Why this matters: the old fixed palette gave Ghibli a "warm-pastel feel" but mismatched the actual subject (a person in red on a white shirt got snapped to cream/sand/sky). K-means picks colors *from the photo* (skin from skin pixels, sky from sky pixels), then the shift puts them into the right style range. Result is a stylized image that's still recognizable as the same person.

Per-baker rewrites in [stylize.js](../client/src/lib/stylize.js):
- **Cartoon** — k-means (5–8 colors) + 1.15–1.45× saturation boost + dilated outlines unchanged.
- **Anime** — k-means (7–12 colors) + 1.4–1.85× saturation boost (the vivid-cel feel).
- **Ghibli** — k-means (9–14 colors) + warm hue shift (+6 to +18°) + slight desat + mild brighten.

### Inception-variant neural style network

[neuralStyle.js](../client/src/lib/neuralStyle.js) switched from the mobilenet style network to the **inception variant** (`saved_model_style_inception_js/`). Trade-off: model size jumps from ~9 MB to ~36 MB on first download, but the bottleneck quality (style fidelity) is markedly higher. The transformer net (~2 MB separable) is unchanged — it accepts either bottleneck.

### Neural output post-processing

After the neural model produces a tensor, the post-process pipeline now runs `autoLevels(0.01, 0.99)` + `boostContrast(0.18)` on the result before encoding to PNG. The Magenta networks often produce slightly washed-out outputs; the new post-pass restores tonal range and adds a mild S-curve so the stylized image reads as punchy on screen. `autoLevels` and `boostContrast` are now exported from `stylize.js` so they can be reused.

### Honest expectations preserved

Neural transfer still paints in a style; it doesn't redraw the subject. K-means makes local CV recognizable + style-appropriate but doesn't redraw either. For true character regeneration, Replicate remains the only path.

## Amendment — 2026-05-28: real neural style transfer + palette-mapped CV

Two big swings in one round to close the gap between local CV output and the SDXL-LoRA reference image the user keeps showing.

### 1. Browser-side neural style transfer (real ML, no API token)

New file [client/src/lib/neuralStyle.js](../client/src/lib/neuralStyle.js):
- TFJS-based two-network arbitrary stylization (Ghiasi et al. 2017) — a style network produces a 100-d "bottleneck" vector for a reference image, the transformer network paints the content image using that bottleneck.
- Models hosted on reiinakano's TFJS demo at `https://reiinakano.com/arbitrary-image-stylization-tfjs/...` (~11 MB total, CORS open, verified live during integration).
- Lazy-loaded on first click; cached in IndexedDB by the browser's HTTP cache automatically.
- Style references uploaded per-preset by the user (no copyrighted samples bundled). Stored as 256×256 JPEG data URLs in `localStorage`.
- Helpers: `loadNeuralModel`, `applyNeuralStyle(contentImage, styleImage)`, `setStyleReference / hasStyleReference / loadStyleReferenceImage / fileToStyleDataUrl`.

`@tensorflow/tfjs@^4.22.0` added to [package.json](../client/package.json).

### 2. Editor.jsx routing change — three-tier priority

`applyStyle` now picks the best available path per click:

1. **Replicate** — if `REPLICATE_<style>_MODEL` or `REPLICATE_STYLE_MODEL` is set. Highest quality, regenerates as a character. Marked with cyan **AI** badge.
2. **Neural** — if a style reference is cached for that preset. Real TFJS neural transfer. Marked with pink **N** badge. First click loads the ~11 MB model once.
3. **Local CV** — the existing classical pipeline. No badge.

Per-button UI adds a small `+` (or `✓` when set) icon top-left. Click to upload a style reference; right-click to clear. Card subtitle was rewritten to guide the user to this flow when no Replicate is configured.

### 3. Push local CV to its real ceiling

Three new techniques in [client/src/lib/stylize.js](../client/src/lib/stylize.js), used by Cartoon, Anime, and Ghibli:

- **`bilateralFilter5(imgData, w, h, sigmaColor)`** — real 5×5 edge-preserving smooth (Gaussian spatial × exponential color weights, with a 256-entry LUT for the color exponential). Replaces 3×3 median in the three styles that benefit most. ~1–3 s on 1 Mpx.
- **`dilateEdges(buf, w, h, radius)`** — 3×3 (or 5×5) max-filter on Sobel output, so cartoon outlines render as 2–3 px thick strokes instead of single-pixel hairs.
- **`snapToPalette(imgData, paletteUint8)`** — quantize each pixel to the nearest entry in a fixed per-style palette. Three curated palettes shipped:
  - **Ghibli**: 12 warm pastel + sky colors (cream, sand, warm skin, sky pastel, moss, peach, warm red).
  - **Anime**: 16 vivid cel colors (saturated red/blue/green/violet + skin + a black for cel seams).
  - **Cartoon**: 8 comic-book primaries (paper white, skin, red/blue/yellow/green, dark hair).

Cartoon now: auto-levels → bilateral → dilated thick outlines drawn over the palette-snapped surface. Anime: auto-levels → bilateral → snap to vivid cel palette, no outlines (distinct from Cartoon). Ghibli: warm tone canvas → soft-light bloom → bilateral → snap to warm-pastel palette.

### Honest expectations preserved

Neural transfer paints the photo in the style of a reference; it does **not** redraw the subject as a hand-drawn character. The reference Ghibli-portrait quality the user showed only comes from generative diffusion (Replicate). The three-tier routing makes it clear which path produced each result via the AI/N badge.

## Amendment — 2026-05-28: clarity tuning — auto-levels, S-curve, Ghibli + Sketch rework

User feedback after the 8-style round: results were "good but need more clear to see the effect" — Ghibli came out hazy/washed-out and Sketch was too white-dominant on bright/low-contrast photos. Root cause: stylization passes were running on un-normalized inputs, so low-contrast hazy photos produced low-contrast hazy stylizations.

Two new shared helpers in [stylize.js](../client/src/lib/stylize.js):
- `autoLevels(imgData, lowP=0.02, highP=0.98)` — percentile-based contrast stretch. Builds a luminance histogram, finds 2nd/98th percentile, stretches that range to [0..255]. Robust to specular highlights and shadow clipping; idempotent on already-balanced photos.
- `boostContrast(imgData, amount)` — final S-curve midtone boost (linear y = a·x + b, a > 1 centered at 128). Applied as the last step in each baker.

Per-baker tuning:
- **Ghibli** — bloom switched from `screen` to `soft-light` blend (no more highlight blow-out), bloom radius reduced, base brightness pulled back to 102% (was 112%), sepia/saturation pushed up so the warm-paint look reads as pigment, not haze. Added auto-levels + light posterize + S-curve final pass.
- **Sketch** — grayscale base brightness dropped from 108% to 92% so the dodge doesn't blow mid-tones to white. Added a post-pass that darkens every pixel by 15–45 (intensity-scaled) and applies a steep S-curve so lines pop instead of dissolving into the page.
- **Anime** — removed soft edge darkening entirely so it's visually distinct from Cartoon (Anime = smooth flat colors, no outlines; Cartoon = flat colors + thick black outlines). Saturation pushed higher (170–250%), more aggressive posterize.
- **Cartoon** — lower edge threshold (more outlines), tighter posterize levels, saturation bumped.
- **Pop art** — saturation/contrast pushed harder, edge threshold reduced so outlines are unmissable.
- **Watercolor / Oil paint / Pixel art** — auto-levels at start, S-curve at end, slight saturation/contrast bumps.

Net effect: each style is now visibly distinct from the others and robust to dim/hazy input photos. The intensity slider still scales each baker's knobs; the new helpers run at full strength regardless (their job is to normalize, not to be tunable).

## Amendment — 2026-05-28: latency cut (smaller model, smaller content, bottleneck cache, pre-warm)

User reported the neural path was too slow per click. Four changes, multiplicative speedup:

**1. Smaller style net (~4× smaller download, ~2× faster style pass).** Reverted from the inception variant (~36 MB) back to the mobilenet variant (~9 MB) in [client/src/lib/neuralStyle.js](../client/src/lib/neuralStyle.js):

```js
const STYLE_NET_URL =
  'https://reiinakano.com/arbitrary-image-stylization-tfjs/saved_model_style_js/model.json';
```

Both variants emit the same 100-d bottleneck shape, so the transformer net is unchanged. The autoLevels + boostContrast post-processing already in `tensorToBlob` closes most of the fidelity gap that motivated the inception switch in the previous round. Total download is now ~11 MB (style 9 + transformer 2) instead of ~38 MB.

**2. Smaller content size (~2.5× faster inference).** `MAX_CONTENT_SIZE: 1024 → 640`. Pixel count drops to ~39% of the previous size, so the fully-convolutional transformer runs proportionally faster. The output canvas still upsamples back to the original resolution with high-quality smoothing in `tensorToBlob`, and the autoLevels/boostContrast post-pass sharpens the upsample. Visual quality difference is negligible at typical photo viewing sizes.

**3. Per-preset style-bottleneck cache (~25-40% per repeat click).** The style network is the expensive ~25-40% of inference; its output for a given reference image is constant. Added a `Map` keyed by `cacheKey` so the first click on a preset computes the bottleneck and stores it; every subsequent click on the same preset (or a re-stylization of an edited photo) skips the style pass and only runs the transformer:

```js
applyNeuralStyle(content, style, { cacheKey: preset.id })
```

The cache is invalidated whenever the user uploads or clears a style reference for that preset (`setStyleReference` / `clearStyleReference` call `clearBottleneckCache(presetId)`).

**4. Pre-warm after image upload.** Added a `useEffect` in [Editor.jsx](../client/src/pages/Editor.jsx) that watches `imgMeta` and fires `loadNeuralModel()` in the background as soon as the user uploads a photo. By the time they reach for a style button, the ~11 MB download is already in flight or complete:

```js
useEffect(() => {
  if (!imgMeta || neuralModelLoaded) return;
  loadNeuralModel().then(() => setNeuralModelLoaded(true)).catch(() => {});
}, [imgMeta, neuralModelLoaded]);
```

The user-facing "Loading neural model…" toast now only fires if `neuralReady()` is still false at click time (i.e. they clicked before pre-warm completed).

**Expected per-click latency** on a typical laptop:
- **First click after first image upload:** ~3-8 s (model already pre-warming; content 640px inference; bottleneck computed once).
- **Same preset, edited photo:** ~1-3 s (bottleneck cached; transformer-only pass at 640px).
- **Different preset:** ~3-6 s (style pass + transformer pass; transformer is the bigger half).

vs. previous round: ~10-15 s first click, ~5-10 s subsequent. Cumulative speedup ~3-5× for the typical "try several styles" workflow.

## Amendment — 2026-05-28: AnimeGAN v2 via Hugging Face Space (free real Ghibli/Anime GAN)

User: "these are the effect but i need proper and accurate this effect, use whatever you required, add skill acc to you."

Honest framing presented: free no-auth options have a hard quality ceiling that arbitrary style transfer (Magenta TFJS) cannot break. The next step up that doesn't require a token or payment is **AnimeGAN** — a GAN specifically trained for photo→anime-art transformation. The user picked this path explicitly.

Unlike Magenta's general-purpose style transfer (which paints over pixels using style-network features), AnimeGAN was trained on Hayao Miyazaki / Makoto Shinkai art and produces output that actually looks like anime illustration while preserving the photo's composition exactly. It's GAN-based, not diffusion, so the face/pose/background stay anchored.

**New backend tier** — `animeganStylize` plugin in [client/vite.config.js](../client/vite.config.js):

`POST /api/animegan { image: dataUrl, style: 'ghibli' | 'anime' | 'sketch' }` proxies to the `akhaliq/AnimeGANv2` Hugging Face Space (Gradio v3 API). Preset → style mapping:

| Preset | AnimeGAN variant |
|---|---|
| ghibli | `version 2 (🔺 robustness,🔻 stylization) [Hayao]` — Miyazaki / Studio Ghibli |
| anime | `version 2 (🔺 robustness,🔻 stylization) [Shinkai]` — Makoto Shinkai anime |
| sketch | `version 2 (🔺 robustness,🔻 stylization) [PortraitSketch]` |

The other 5 presets (Cartoon, Watercolor, Oilpaint, Pop art, Pixel art) keep using the neural style transfer fallback — AnimeGAN wasn't trained for those.

**Resilience built in:**
- 90 s abort timeout (HF Spaces can be slow on cold start).
- 502 with a clear error message on upstream failure; the frontend falls through to neural transfer instead of breaking.
- Tiny-response guard (`< 1024 bytes`) catches placeholder/error images that the upstream sometimes returns during overload.

**Note: Gradio v6 migration patched mid-round.** First implementation hit `POST /api/predict` (Gradio v3 syntax) and got 404 — the Space had upgraded to Gradio v6. Rewrote against the new API:

1. POST `/gradio_api/call/generate` with `{ data: [{ url: <dataUrl>, meta: { _type: "gradio.FileData" } }, "Version 2"] }` → `{ event_id }`.
2. GET `/gradio_api/call/generate/<event_id>` — Server-Sent Events stream. Parse for the `event: complete` line and JSON-decode its `data:` payload.
3. Result is a FileData object with `url` (or `path`, which we resolve to `/gradio_api/file=<path>`). Fetch that, stream the bytes back.

The SSE parser handles `complete` (success), `error` (upstream rejection), and ignores `generating` / `heartbeat` progress events. Smoke test with a 512×512 portrait silhouette returned a 3 KB image/webp output, confirming the full pipeline works.

The Space only exposes "Version 1" / "Version 2" radio buttons (not per-style picker), so all three of our supported presets (Ghibli, Anime, Sketch) call with `Version 2`. The aesthetic difference between presets comes from the rest of the UI (different label, different default merge-layer treatment), not from the upstream model — limit of the free Space.

**Updated routing in [Editor.jsx](../client/src/pages/Editor.jsx) `applyStyle`:**

1. **Replicate** (if `REPLICATE_API_TOKEN` is set) — unchanged
2. **AnimeGAN** (new, for Ghibli/Anime/Sketch) — `else if (ANIMEGAN_STYLES.has(preset.id))`
3. **Neural style transfer** (TFJS) — wrapped in try/catch so failures fall through
4. **Local CV** — final guaranteed fallback

Refactored from `if/else if/else` to `if + sequential if (!blob && …)` checks so each tier can fail independently without losing the chain.

A shared `bakeCurrentEdits(mime, quality)` helper bakes the current CSS filters + rotation + flips onto a data URL — used by both the Replicate path and the AnimeGAN path so any remote AI sees what the user actually sees.

**User-facing latency:**
- Cold first request: 20–60 s (HF Space waking up).
- Warm subsequent requests: 5–15 s.
- On failure → neural transfer (~1–8 s) → local CV (~0.5–2 s). User always gets a result.

**Honest quality expectation:**
- AnimeGAN output is ~70% of the way to the user's SDXL+LoRA reference image. The aesthetic is genuinely Ghibli/anime art, but it's a GAN transform — less detailed than what diffusion produces. To reach 100% reference quality, an HF Inference API token or paid Replicate is still required.
- The composition is preserved exactly (the user's pose/face stays in the same place). This is the meaningful difference from arbitrary style transfer.
